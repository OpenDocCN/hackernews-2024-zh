# **隐形的通才 | 阿道夫·奥查加维亚**

> 来源：[https://ochagavia.nl/blog/the-undercover-generalist/](https://ochagavia.nl/blog/the-undercover-generalist/)

<content>自从成为独立承包商以来，我一直感到作为通才软件工程师，却不得不将自己营销为专家之间存在一种紧张关系。多年来，我一直想写一写这件事，甚至为此保留了一些笔记。最近，我看到了[一篇文章](https://social.clawhammer.net/blog/posts/2024-01-19-CultureChange/)，作者是[本·柯林斯-萨斯曼](https://www.red-bean.com/sussman)，这给了我最后一点灵感，尽管他的文章只是间接触及了这个话题。

下面是我挣扎的经历，希望对其他冒险者有所帮助。请注意，我的话语与我的经验紧密相关，因此请谨慎对待！

## 需要看起来像专家

我认为一位HN用户的以下表述非常精准：“理论上，所有公司都喜欢适应性强的人。实际上，我发现大多数职位描述更偏爱专家”（[来源](https://news.ycombinator.com/item?id=32921907)）。这意味着，即使你是通才，当你寻找工作时，你可能也需要将自己表现为专家！

我花了一些时间才接受这个现实。自从[我开始摆弄电脑](https://ochagavia.nl/blog/how-i-got-into-programming/)，我从未感到有必要将我的技能框架化为特定的类别。我认为自己是一个“黑客”，按照[术语文件](http://catb.org/jargon/html/H/hacker.html)的意义，这对我来说已经足够了。我甚至有幸在一家与我分享这种观点并积极聘请充满好奇心的通才的公司开始我的编程生涯。那段时光真是太棒了！

然而，当我成为独立承包商后，作为通才的表现开始显示出其局限性。潜在客户想要听到一些符合他们思维范畴的东西，而不是像“我是计算机奇才”这样模糊的说法。他们会问一些问题，比如：你是前端还是后端工程师？你是用.NET还是Python编程？你是专注于AWS还是Azure？等等。

我不怪他们……我猜，当你需要（或至少*认为*你需要）某一组特定技术的专家时，评估雇佣通才问题解决者的价值是困难的。虽然我确实成功找到了一些重视我作为通才的客户，但我知道我的沟通策略需要改变。

## 对人们说他们想听到的话

到了2022年底，我开始将自己定位为专家：一个专注于系统编程和开源开发的Rust专家开发者。不管是幸运还是天赋，这种策略立刻奏效，并在整个2023年保持了合同的持续签署！

这种品牌策略的选择受到了多种因素的驱动：

+   我自2014年以来一直[参与Rust社区](https://ochagavia.nl/blog/how-i-got-involved-in-the-rust-community/)，尽管我不是著名的Rust人物，但我可以自信地说我深入了解这门语言。

+   参与Rust社区为我带来了重要的开源经验，我非常喜欢。寻找开源工作似乎是理所当然的选择。

+   我喜欢开发我所称的“基础软件”（创造构建模块，而不是粘合API），而Rust通常用于这一目的（如果你在写CRUD，我认为使用其他语言和生态系统会更好）。

+   我真心喜欢Rust这门语言，尽管它并不完美。

+   Rust在工业界的使用在过去十年中一直在[稳步增长](https://www.tiobe.com/tiobe-index/rust/)，因此继续专注于它似乎是一个明智的选择。

## 那么…… 你*是*变成了专家了，对吗？

从外表看，可能会这样认为，但一旦深入了解，你会发现我其实还是一位全才。

首先，我参与的每个Rust项目都要求我广泛的技能和黑客思维。我不得不投身于全新的问题领域和代码库，唯一的共同点是这些项目都是用Rust实现的。以下是一些例子：

+   改进[Quinn](https://github.com/quinn-rs/quinn/)，社区制作的Rust QUIC协议实现：我实现了一个缺失的可选功能（以及[this RFC draft](https://datatracker.ietf.org/doc/html/draft-ietf-quic-ack-frequency-04)）。

+   [为Python / Conda生态系统开发软件包管理器](https://ochagavia.nl/blog/the-birth-of-a-package-manager/)：我编写了[依赖解决器](https://ochagavia.nl/blog/the-magic-of-dependency-resolution/)，最终成为[resolvo](https://github.com/mamba-org/resolvo)库。这是我职业生涯中第一次因阅读学术论文而获得报酬。感觉有点像回到大学…… 但是这很棒！

+   为[rustls](https://github.com/rustls/rustls)，流行的Rust TLS协议实现，创建了一个性能基准测试系统：我从Rust编译器的基准测试方法汲取灵感，并经过大量研究，为rustls[实现了一个类似的系统](https://ochagavia.nl/blog/continuous-benchmarking-for-rustls/)。

除了我参与的多个Rust项目，还有一个让我仍然认为自己是全才的额外原因，即我也在做完全无关的事情。例如，以下是我正在进行的专业项目：

+   开发并维护一个桌面软件包，供非技术化学团队使用（C# / .NET）。

+   开发并维护一个网页应用程序，帮助一个非营利组织进行记账（C# / .NET）。

+   担任一家初创企业的领导岗位，每周工作一天（使用Python技术栈，尽管我几乎不编程）。

## 到目前为止，我的结论是

矛盾的是，似乎要获得“通用”项目，首先必须表现自己是一个“专家”！这怎么可能呢？

让我们从我提到的非Rust项目开始。所有这三个项目都是通过认识我的人传递过来的，他们寻找一个“可信赖的计算机专家”。这些人对我的互联网存在一无所知，只是想做点什么，不管底层技术如何。从这个角度来看，我认为作为专家的品牌对此没有任何影响。他们关心的只是我们互相信任，我擅长软件工程。

然而，对于Rust项目而言，品牌确实产生了影响。顾客们之前不太了解我，他们希望有些保证我能够交付。看来我的“Rust身份”充当了信任的源泉。他们能看到开源维护者们重视我的贡献，我曾为这门语言撰写过RFC，我得到了生态系统中受尊敬的人士的推荐，等等。

综合考虑这一切，我的当前理论是，专注于特定技术的经验以及在特定社区的参与，可以更容易地与不太了解你的人建立信任。随着信任的增长，揭示出潜在的通用性更有可能被接受！
