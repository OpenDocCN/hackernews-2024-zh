# 解读 Glyph :: 软件需要更贵

> 来源：[https://blog.glyph.im/2024/03/software-needs-to-be-more-expensive.html](https://blog.glyph.im/2024/03/software-needs-to-be-more-expensive.html)

## 咖啡的成本

在咖啡行业中，[詹姆斯·霍夫曼](https://www.youtube.com/@jameshoffmann) — 或许是最有影响力的…… 影响者 — 努力推广的一个理念是“咖啡需要更贵”。

咖啡行业以剥削性而闻名。尽管独立咖啡馆老板的利润空间相对较小，[劳工剥削](https://www.theguardian.com/business/2020/mar/01/children-work-for-pittance-to-pick-coffee-beans-used-by-starbucks-and-nespresso)甚至[奴隶制](https://www.dol.gov/agencies/ilab/addressing-child-labor-and-forced-labor-coffee-supply-chain-honduras)在咖啡供应链中却屡见不鲜。

简而言之，霍夫曼先生在一系列长视频和采访中提出的一个观点是（一些）可以通过监管努力来解决。制造商和政府对供应链政策的执行可以帮助发现并避免这类剥削。消费者的辨别力也可以解决一些问题。你可以尝试购买公平贸易咖啡，避免那些你知道有问题的供应链历史的品牌。

但最终，即使供应链的完美、普遍和零成本的执行…… 消费者仍然必须愿意，你知道，*为咖啡付更多的钱*。支付工资比拥有奴隶成本更高。

## 软件的价格

咖啡供应链问题本身值得引起重视。我并不是说开源维护者的问题与儿童奴隶的问题一样严重。但原则是相同的。

每个科技公司都大量使用免费的开源软件。

我不想简单地说这是剥削。开源维护者在这里有一个复杂的交易：如果你创建开源软件，你可以更容易地[找到工作](https://opensource.com/article/19/5/how-get-job-doing-open-source)。如果你创建开源基础设施，你可以在技术上做出更具长期可持续性的架构选择，但在较短期的商业开发时间表上可能更难以证明合理。你可以与行业内更广泛的群体合作。你可以建立个人品牌。

但是，考虑到最近的[xz Utils / SSH后门丑闻](https://boehs.org/node/everything-i-know-about-the-xz-backdoor)，很明显，虽然这个交易可能并不完全一边倒，它却*不*对称，可能会造成重大的不良后果，对维护者本人和整个社会都是如此。

要解决这个问题，开源软件需要变得更贵。

开源的一个*重要*吸引力部分在于其隐含的许可结构，这既源于其零前期成本，也源于其零边际成本。

零前期成本意味着你可以随意试用它。在许多公司，个人软件开发人员没有权力写采购订单，甚至不具备小额支出的公司信用卡。

如果你是一名软件工程师，需要一个新的开发工具或者想要购买一本新的图书馆供工作使用，那么要获得批准可能会是一场官僚主义的迷宫。这可能是*可能*的，但你可能会得到奇怪的眼神，而且很可能有人，很可能是你的经理，会说“这件事难道没有免费的选项吗？”在最坏的情况下，这可能根本不可能。

这是有道理的。处理采购订单和报销请求很烦人，只有在你处理足够大块的功能时，整个团队或者更好的是整个组织才值得这样做。这意味着大多数采购是由管理人员或“架构师”完成的，他们有权为更大的团体做出决策。

当个人工程师需要解决问题时，他们会特别看重开源库和工具，因为这样很快、很容易将它们整合到一个拉取请求中，而专有解决方案可能会很繁琐和昂贵。

这是假设你的问题的专有解决方案甚至存在。在软件经济的基础设施部门，来自操作系统提供商（Apple、Microsoft，也许是Amazon如果你在云端）和开源开发者的免费选项已经将小型商业选项边缘化或直接摧毁了，因为这个原因。

如果零前期成本是减少文书工作的好处，那么零*边际*成本几乎是一种要求。开源维护者的一个长期抱怨是，公司拿走我们的东西，将其构建成产品，然后赚取大量利润，却没有给我们任何东西。他们给我们一些版税似乎是公平的，对吧？但一旦你意识到个人开发者没有权力在公司卡上刷50美元购买一个工具，他们*绝对*没有权力做出一项技术决策，*使公司核心产品的知识产权附加负担*，以将公司收入的某个部分捐给第三方。从结构上看，这永远不会发生。

尽管存在这些障碍，维护这些依赖关系确实是需要花钱的。

## 已经存在一些解决方案

已经有各种官方渠道正在发展，以帮助支持关键基础设施的维护。如果你在一家大公司工作，你可能应该考虑订阅公司的[Tidelift](https://tidelift.com)服务。也许可以向你的雇主询问一下。

但是，[正如他们乐意承认的](https://social.coop/@luis_in_brief/112185864683054685)，即使是Tidelift也无法覆盖*很多*项目，没有官方商业支持，短期内也无实际方式提供支持。像我这样的个人维护者，试图想出如何通过使其成为谋生手段或将其纳入我们的工作中来维护他们的项目。那些有Ko-Fi或Patreon的人，或者可能只是有一个亚马逊心愿单，让你可以感谢他们偶尔的维护工作。

最重要的是，对他们来说没有*过渡*到实际靠维护工作谋生的途径。对于大多数维护者来说，Tidelift支付的金额低得像副业一样，甚至建立起来（以及GitHub赞助等）也是一个巨大的麻烦。因此，即使从“没有收入”过渡到“一点点副业收入”可能也是过于繁琐。

让我举自己为例。如果你是一个开发者，在你的职业生涯中名义上从[我的基础设施工作](https://pypi.org/user/glyph/)中获益，那么你很有可能也是对开源共享贡献甚至比我多的人，对我的职业成功的贡献甚至比我对你的多。我可以要求你支付给我^(，但实际上*不应该是你支付给我，而应该是你的雇主。

## 现在该做什么：让支付变得简单

所以，如果我们只需给开源维护者更多的钱，而真正应该给钱的是雇主们，那我们能做什么？

让事情简单些。雇主们应该*只是给维护者们钱*。我们称之为“JGMM”福利。

具体而言，每个软件工程师的雇主应立即推行以下福利计划：每个软件工程师每月应有$50的自由预算，用于按照他们认为合适的方式分配给*任何开源依赖开发者*。无论是Venmo、Patreon、PayPal、Kickstarter还是GitHub赞助，都无所谓。把它放在公司卡上，项目名称放在账单上，然后就忘了它。这仅用于开源维护，但金额足够小，无需严格的批准流程。可以靠诚信制度来执行。

这保留了零前期成本。要开始使用一个依赖项，你仍然只需使用它^(。它也保留了零边际成本：你的开发人员根据需求和流行度选择支持哪些依赖项。这是一个固定的开销，不会随着收入或利润而扩展，只是按人头计算。

因为这里的整个重点是*匹配*大多数公司中*可以添加依赖*的简单、隐性、无流程、无控制方式。支付这些小费应该比首次使用软件更容易。

这不到1%的人工成本开销将大幅降低您使用的开源项目的风险。通过将决策权留给您的工程师，您将最终支持那些真正陷入困境、您的高管甚至不会听说直到它们登上新闻的项目。其中一些资金将用于您不使用的项目，这些是您的工程师觉得很有趣并希望将来使用但目前还不依赖的项目，但这也没问题。将其视为极为廉价、高效的研发开支。

开发人员支持开源基础设施的选项大部分已经可以享受税收减免，因此，如果他们为像[PSF的财政赞助人之一](https://www.python.org/psf/fiscal-sponsorees/)这样的项目做贡献，可能比普通业务支出更有税收优势。

我也非常确信，如果您是首批这样做的雇主之一，您可以在技术媒体上获得一轮真正积极的公关报道，并吸引工程师，所以比赛已经开始。虽然我不算“技术媒体”，但无论如何，请[给我来信](mailto:oss-support-471610@glyph.im)，让我知道如果您的公司最终采取了这种做法，我可以为您大声喝彩。
