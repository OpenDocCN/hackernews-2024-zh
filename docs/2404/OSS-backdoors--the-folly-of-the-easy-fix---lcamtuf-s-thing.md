# OSS backdoors: the folly of the easy fix - lcamtuf’s thing

> 来源：[https://lcamtuf.substack.com/p/oss-backdoors-the-allure-of-the-easy](https://lcamtuf.substack.com/p/oss-backdoors-the-allure-of-the-easy)

在[昨天发布的文章](https://lcamtuf.substack.com/p/technologist-vs-spy-the-xz-backdoor)中，我深入探讨了一个远程代码执行后门在一个称为*liblzma*的压缩库中的详细情况。这个后门可能是计算机安全史上对开源生态系统最复杂（也是最大胆！）的攻击之一。

当故事曝光后，一些评论员急于归咎于未支付的维护者与受益于其工作的公司之间不健康的关系。我觉得这个论点不具有说服力：

> *“许多小型基础 OSS 库的真正问题只是没什么可做。它们是几十年前由一个人编写的 — 除了修复错误外，它们真的不应该有太大变化。你不会每年对 zlib 或 giflib 进行重大更新；即使你挥舞一些现金，也很难围绕观看干漆建立可持续的社区。”*

此后，一批新声音 — 这次是来自科技公司 — 开始呼吁开源维护者加强项目治理。其中一些提议包括强制两人代码审查、自我评估、服务水平协议和书面继任计划。

我不确定我同意。轻轻地推动生态系统朝正确的方向发展是可以的 — GitHub 可以做到！ — 但俗语说得好，你不能超越摩萨德。这不是一种失败主义的立场：只是我们有专门化的原因。*libcolorpicker.so* 的维护者不能是你的关键基础设施和俄罗斯或中国情报机构之间的唯一阻挡。间谍会被间谍阻止。

即使涉及较小的威胁，底线是我们在由业余爱好者开发的代码之上有数以万亿计的资金。从这些基础设施中获利的公司可以负担得起代表社区彻底审核和监控关键依赖。明确地说，一个全面的解决方案将是一项困难且昂贵的任务 — 但它并不比大语言模型或自动驾驶汽车更难或更昂贵。

现在我们已经被警醒，答案不是依赖 John Q. Maintainer 并要求他们下次做得更好。答案是解决任何反情报工作上的失误，并大大提高幕后检测能力。
