# 电子邮件转发的终结日渐临近 - TidBITS Talk - TidBITS Talk

> 来源：[https://talk.tidbits.com/t/the-demise-of-email-forwarding-is-getting-closer/27315](https://talk.tidbits.com/t/the-demise-of-email-forwarding-is-getting-closer/27315)

这个问题实际上并不影响你所描述的直接将某条消息转发到另一个邮箱地址的方式。当你这样做时，你的电子邮件地址会被应用为发件人。这基本上与直接发送消息一样工作，除非你有自定义域名，否则你可能不需要太担心会出现问题。

首先面临风险的是最初自动的服务器级转发方式。当你以这种方式转发消息时，它会保留原始发件人信息。这个发件人的域名与消息当前正在通过的域名不匹配。SPF、DKIM 和 DMARC 是一组用于验证消息是否合法的协议，尽管域名不匹配。

为了继续这种转发，消息经过的每个域名都需要正确实施 SPF、DKIM 和 DMARC。转发域名可以更新以实施这些措施，但被转发的域名（比如你的旧校友邮箱域名）也需要实施这些措施，而许多这些较老的域名不再积极管理。它们仍然注册着（否则，发送到它们的电子邮件会立即退回），但如果它们实际上除了维持转发之外不再使用，它们很可能不会费心实施必要的协议，你可能也联系不到管理这些域名的任何人来更新它们。

你的 Microsoft 邮箱可能不会有任何问题。虽然 Microsoft 目前似乎还没有要求发送的邮件使用 DMARC，但对于接收到的邮件却应该实施了。因此，如果他们管理着那个旧的校友邮箱域名，他们可以确保 DMARC 被正确设置。

很大程度上取决于你接收邮件的域名。如果是 Google，他们正在拒绝所有未经 DMARC 认证的消息（如果这对于所有收件人尚未实施，很快也会实施）。其他主要的电子邮件提供商正在各个阶段实施对入站邮件的 DMARC 要求。

通常你不会知道发给你的邮件被退回了，因为退信会发送给发件人。只有当你知道应该收到某个消息却没有收到时，你才可能注意到。因此，你期望转发给你的消息会因为这些旧域名未更新设置而消失。

如果您向一个拒绝非规范消息的服务器发送非规范消息，您会收到退信。这基本上只有在您使用自定义域名时才有风险，这种情况下，您需要按照[另一个主题](https://talk.tidbits.com/t/need-to-check-dmarc-settings-for-custom-domains)中的指示确保您的域名正确设置了协议。这并不难，但您的域名托管商不会无形地为您执行这些操作。每个域名所有者/管理者都有责任在其域名上设置 SPF、DKIM 和 DMARC。
