# 查找和修正有关程序行为标准误解的标准。

> 来源：[https://blog.brownplt.org/2024/04/12/behavior-misconceptions.html](https://blog.brownplt.org/2024/04/12/behavior-misconceptions.html)

*发布于2024年4月12日。*

现代语言中有大量共同的语义核心，从Java和C#到Python和JavaScript，再到Racket和OCaml：

+   变量在词法上有作用域

+   作用域可以嵌套

+   评估是急切的

+   评估是顺序的（按“线程”）

+   变量是可变的，但是是一阶的：

+   结构（例如，向量/数组和对象）是可变的，并且是一阶的

+   函数可以是高阶的，并且可以闭合词法绑定

+   内存是自动管理的（例如，垃圾回收）

我们称之为*语言的标准模型*（SMoL）。

**SMoL**潜在地具有巨大的教育价值：

+   如果学生掌握了SMoL，他们对这些语言的核心有了很好的把握。

+   学生可能会发现在不同语言之间移植他们的知识更容易：他们可以在共同的语义特征中找到熟悉的路标，而不是迷失在不同语法的海洋中。这也可能使学习新语言更容易。

+   不同语言之间的*差异*变得更加明显。

+   学生可以看到，通过超越语法，有几个重要的语义概念贯穿所有这些语言，其中许多被我们认为是编程语言设计中的“最佳实践”。

因此，我们过去四年一直致力于SMoL的教学法：

+   查找SMoL程序行为理解中的错误。

+   查找这些错误背后的（误解）概念。

+   将这些整理成干净的工具，易于部署。

+   构建一个*导师*来帮助学生纠正这些误解。

+   验证上述所有内容。

现在我们准备展示这一努力的检查点。我们已将这项工作的精髓提炼成一个工具：

[SMoL导师](https://smol-tutor.xyz/)

它能够识别并尝试修正学生的误解。该导师假设用户具有通常在1-2门课程后找到的编程知识基础：变量、赋值、结构化值（如向量/数组）、函数和高阶函数（`lambda`）。与大多数导师不同的是，它不是*教授*这些概念，而是调查用户对它们的理解程度。无论用户在哪里犯错，导师都使用称为*反驳文本*的教育工具帮助他们理解错误所在，并纠正他们的理解。导师允许用户在多种语法之间切换，这样他们可以选择最舒适的一种工作方式（以免语法上的陌生感或不适成为错误的源头），并且可以看到这些语言之下的语义统一性。

为了更好地分类学生的回答，我们提出了一个称为*misinterpreter*的概念。一个*misinterpreter*是一个有意错误的解释器。具体来说，对于每个误解，我们都创建一个相应的*misinterpreter*：在大部分方面它与SMoL具有相同的语义，**除了**在那个误解特征上，它实现了误解而不是正确的概念。通过使误解可执行，我们可以机械地检查学生的回答是否对应于误解。

这里有许多有趣的教训：

+   许多问题程序对专家来说可能会显得非常简单。

+   对于学生来说，状态、别名和函数的组合非常复杂。（然而，大多数*入门*课程毫不犹豫或关心地将学生投入这些主题的漩涡中。）

+   *Misinterpreters*本身是一个有趣的概念，并且可能独立于上述用途具有价值。

此外，我们尚未直接研究以下主张，但基于对这项工作和教授和讨论编程语言的经验的观察，我们认为它们是完全合理的：

+   在SMoL语言中，局部和顶级绑定的行为与函数调用引起的绑定相同。然而，学生们经常没有意识到这些具有统一的语义。部分原因可能是我们对“按调用”术语的关注，它专注于调用（并使它们看起来很特殊）。我们认为用“bind by”取代这些术语会有所改进。

+   我们也认为，“按值调用”和“按引用调用”这些术语在目前（学生、教师、博客、网络等方面）已经混淆不清，因此总体上找到更好的术语将会有所帮助。

+   我们非正式地讨论编程概念（例如“传递一个变量”）的方式，以及我们语言所做的语法选择（例如`return x`），几乎肯定也是混淆的根源。前者自然地会使学生认为变量被别名，而后者会使他们认为返回的是*变量*而不是它的*值*。
