# Filippo的开源维护政策

[](#filippos-open-source-maintenance-policy)

链接：[https://filippo.io/maintenance](https://filippo.io/maintenance)

更新日期：2024年4月

许可证：[知识共享署名4.0国际许可证](https://creativecommons.org/licenses/by/4.0/)

本政策解释了我在开源项目中的工作方式，作为维护者，你应该期望我做些什么，以及如何做出贡献。

我参与了许多代码库的工作，而适合我的大脑的最佳关注方式是批量处理工作，专注于一个项目一到四周，然后转移到另一个项目。这减少了上下文切换的开销，并给我时间考虑整体趋势。

这意味着可能会有较长时间我并未积极地在特定项目上工作。这并不意味着项目被放弃：我依然会*阅读*每一个提交的问题，并对紧急问题作出及时反应，例如安全报告或是使项目对*当前*用户不再有效的故障。

如果一个项目不再维护，它将被归档或移至[FiloSottile/mostly-harmless](https://github.com/FiloSottile/mostly-harmless)。

### 项目有明确的范围

[](#projects-have-a-precise-scope)

我的大多数项目旨在成为特定问题的简单解决方案，我通常会尽量抵制范围的扩展。

我理解这可能会使它们不适合你，这没关系。请随时开启讨论或提出功能请求，但请理解如果我们得出结论认为问题超出了范围。欢迎分支！

这对于保持用户体验简单和减少维护负担都至关重要。

### 精心研究过的问题就像黄金一样宝贵

[](#well-researched-issues-are-gold)

最有用的贡献是详细的问题报告。你能告诉我们环境和期望的越多，越好。当用户进一步调查或检查其他实现方式或相关规范时，我真的很感激。

即使你不能做额外的工作，也可以放心地提出问题。我不会厌恶任何问题，它们都是礼物（但作为礼物，它们并不意味着你有权得到回复）。

### 经验报告就像宝石一样珍贵

[](#experience-reports-are-gems)

即使某些东西看起来按预期工作，或者您遇到非技术性问题，我还是建议您提交一份经验报告。

可以简单到“这个错误让我困惑，即使我弄清楚了”，也可以高层次到“起步混乱，这是我尝试的所有事情”，或者具体到“当我尝试在这个工作流程中使用它时非常笨拙”。

焦点应放在您的经验上，而不是解决方案。通常解决方案需要考虑其他利益相关者的需求，或者同时解决多个问题，但明确问题的想法是第一步。

也许我们会得出结论，在项目的范围内（见上文），我们无能为力，但每一份经验报告都会影响我对项目的思考。

如果你想贡献但不知从何处开始，请尝试从头开始使用该项目，记录所有让你感到困惑、困难或不清楚的摩擦，并分享出来。

### PRs 可能会被重新实现。

[](#prs-might-get-reimplemented)

与一些项目不同，出于多种原因，我*不*偏好 PRs 而不是问题。

+   批量工作（见上文）意味着你可能需要等一段时间才能得到我的 PR 审查，到那时你可能已经转向其他工作。

+   我没有时间进行漫长的审查周期，但我对接受项目中的代码也非常挑剔，因为后续维护工作将由我来负责。

+   密码学和安全项目需要异常高程度的代码审查。

+   贡献者在编写补丁时自然会关注他们的用例和情况，这可能需要调整以服务其他项目用户。

+   我通常不会试图在大多数项目周围建立大型贡献者社区，因为它们是小范围工具（见上文）。

总之，详细的问题对我通常比 PRs 更有用，如果你想展示问题，欢迎提交 PR，**但请预期它可能会被重新实现而不是合并**。请不要个人化对待。我尽量在合作者中自由使用 `Co-authored-by` 来分享功绩。

也有一些例外情况，比如我不熟悉的操作系统的支持，或者我不擅长实施的功能。我们可以在问题跟踪器中讨论这些情况。通常我也很乐意合并微不足道/简短的更改。

我是一个[专业的开源维护者](https://words.filippo.io/professional-maintainers/)。如果您的公司可能有兴趣支持我的工作，在[与维护者直接联系的相互价值](https://words.filippo.io/dispatches/reciprocal/)和无限访问我的专业知识方面，请联系讨论保证金！📨

*我相信开源维护者的关键角色可以发展成为一个真正的职业，其薪酬应与高级软件工程师相当，向依赖开源项目并希望获取维护者独特和广泛专业知识的公司收费。*

**我不依赖于众筹来维持，因此我通常不向个人募捐**。如果您想表达感激或鼓励我的工作，请给我寄明信片！📮

我的大多数项目没有正式的行为准则，但我秉持[Go社区行为准则](https://go.dev/conduct)的价值观，并且对有毒行为持零容忍政策。

创造不友好环境的评论将会被封禁。这是我的领地，你可以去别处发牢骚。

请私下发送电子邮件至[security@filippo.io](mailto:security@filippo.io)来报告潜在的安全问题。如果你不确定某事是否是安全问题，请联系我！你应该在一周内收到答复，通常会更快。

我愿意接受标准的90天披露时间线，或者不超过9个月的禁令期。我将为任何我认为是安全问题的问题发布安全公告并申请CVE编号。

我知道项目没有免费的安全研究或协调的披露权利，我感谢报告者的贡献。我目前不提供赏金计划。

我*不会*仅仅为了消除与我的项目无关的漏洞自动报告而更新依赖项。（我也不认同无休止的依赖关系更新，尽管我对此的看法更加微妙。）

使用自动化工具，这些工具甚至未能进行包级别的范围限定，然后将无用信息推送给每一个上游项目，这是不可持续的。我的责任是确保我的项目不受安全漏洞的影响。扫描工具的责任是确保它们不会因误报而打扰用户。

[govulncheck](https://pkg.go.dev/golang.org/x/vuln/cmd/govulncheck)非常棒，甚至可以进行静态分析以在符号级别过滤漏洞，但是你应该要求你的供应商进行简单的包级别过滤。
