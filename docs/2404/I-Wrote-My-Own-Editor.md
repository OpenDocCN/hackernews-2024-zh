# 我写了自己的编辑器

> 来源：[https://bold-edit.com/wrote-my-own.html](https://bold-edit.com/wrote-my-own.html)

<main class="wrote-my-own">

我曾经喜欢 vscode。自从它可以运行和调试 C# 以来，我一直在使用它。去年，我注意到我经常遇到不能输入的 bug，直到我用 alt-tab 切换出去再切换回编辑器。这并不算太糟糕，但当我不能构建或调试直到我重新启动程序时，这就很烦人了。

有一天，可能最让我困扰的事情是我在写一个 GDB 的漂亮打印机时。我调整脚本；然后为了查看结果，我会按 F5 运行已经构建好的二进制文件，每次都要等待超过一秒钟（大约 1480 毫秒）。这让我疯狂。最后我决定保持一个终端打开，保存后，切换窗口，按上箭头，回车。这样做起来仍然比在 vscode 重新启动调试器快了几乎一秒钟。我意识到，无论我的编译器有多快（我为自己的语言实现了一个），如果我使用 vscode 的话，它总会感觉很慢。（编译器在 15 毫秒内生成一个可调试的二进制文件。）

最后，让我无法忍受的是当我在调试三个不同的复杂函数时。每个函数大约有 15-20 个我感兴趣的变量。我让监视面板占据整个右侧，但空间还不够。每次我切换到第三个函数时，我都不得不**手动删除和添加变量**。我并不总是按相同顺序重新放置这些变量。这太烦人了。无论我看的是哪个函数，我都希望每次右上角看到相同顺序的相同一组变量。在遇到 bug、启动速度慢，并且调试变得成问题后，我当时决定写一个文本编辑器。

我想，这有多难呢？答案总是比你想象的要难，但我知道这一点，我想找出答案

我做的第一件事是实现 GDB 支持。这是一个视频，显示了右上角的观察变量根据其所在的函数而改变

<wrote-my-own-vids/dynamic.mp4>

如果你仔细看，你会发现没有闪烁。所有变量及其值同时显示。你可以看到变量显示得非常快。从视频中你看不到我按下按钮启动调试器的时刻。从启动到断点（包括构建步骤）仅花了 <200ms。我解决了我的两个最大的烦恼。接下来，我想确保无论文件大小如何，编辑和渲染都是快速的。有些工具生成数百兆字节的 JSON。这是一个视频，展示了我打开两个文件（都是 1gb），滚动，打字，以及撤销/重做的过程。

<wrote-my-own-vids/large.mp4>

我很高兴我可以在调试版本中保持 60 帧每秒的运行而没有丢帧（vscode 会丢帧）。但是对于 4K 显示器，我需要一个优化的版本。即使在打字和撤销/重做时，我也一直保持在 144 帧每秒以上，这样有快速显示器的人可以有一个流畅的体验。

我喜欢能够部分输入文件名并跳转到它，或查找/跳转到类型。我喜欢重命名变量。LSP（语言服务器协议）支持这些功能，所以我实现了它。这不是完整的实现，一个容易看到的问题是没有语义标记支持，这会改进高亮显示。视频展示了签名帮助，但没有展示跳转、重命名、悬停等功能。朋友们不喜欢在他们当前的 IDE 中调试，所以他们问我是否能实现 DAP 支持。所以我也实现了这个。视频展示了一个使用 Python 的例子。

<wrote-my-own-vids/lsp.mp4>

几周前，我开始写作之前，我遇到了工作在[撤销调试器](https://undo.io/)上的人们。他们的目标是让人们可以重放执行过程，这样你就可以找出和调试那些没有人知道如何重现的复杂问题。但这不是我喜欢它的原因。我做了这个。

<wrote-my-own-vids/udb.mp4>

在那个视频中，我对全局变量 g1 运行了 'last' 命令。调试器将我带到了它最后修改的时间点。这简直是魔法。即使我知道它在哪里被修改，我也经常使用它。它使我免于在断点上写条件语句，而且如果不知道变量最后一次修改的位置，我也不必猜测。它不是一个免费的调试器，但我想为自己实现这个功能，而且它非常容易，因为它与 gdb 具有相同的协议，实现只是在我已有的 gdb 实现之上添加了几行代码。

这个编辑器仍处于早期开发阶段，还有很多功能缺失。没有全局搜索，只有查找功能但没有替换功能，喜欢 vim 操作的人需要等待。如果你试用了 5 分钟，请随时告诉我你注意到的所有缺失的功能。下载文件中有一个教程文件，可以在使用这个编辑器的前几分钟内做些事情。
