# 猴子管理 | 米希尔编年史

> 来源：[https://mihirchronicles.com/monkey-management/](https://mihirchronicles.com/monkey-management/)

1974年，《哈佛商业评论》发表了由威廉·昂肯和唐纳德·沃斯合著的文章《谁拿到了猴子》，描述了员工委托上升的未解决问题，导致“猴子”落在经理的身上。这增加了经理的工作负担。

这个类比对我影响深远。我在这个类比上加了自己的理解，并将其应用于我的产品管理世界。产品经理并不拥有团队，但他们需要在横向、纵向和下向管理中出色。如果我们在猴子管理上做得不好，很容易失去时间和个人自主权。对于工程、设计和领导来说，指责产品团队成为瓶颈会更容易。

让我们探讨一个场景，展示猴子如何从一个部门跳到另一个部门。

领导团队决定致力于一个关键战略倡议。该战略对整个组织广播，并由产品团队负责执行。他们需要设计和工程的帮助。设计部门将探索解决方案并帮助理解*解决方案的可用性*。工程部门将探索*解决方案的可行性*。需要在业务的其他功能之间进行协调。但为了简单起见，我们就到此为止。

产品团队提出了一个问题陈述，说明需要完成的任务。设计部门提出了解决方案。工程部门认为提出的设计存在问题，并要求产品重新考虑需求。

让我们分析问题是如何从一个部门传递到另一个部门的。

领导层将战略执行任务传递给产品部门。这是由上级强制的，意味着产品团队不能忽视。预期需按时交付，否则团队将面临惩罚。

在专家知识基础上确定需求后，产品将问题传递给设计部门。这是系统强制的，意味着需要跨职能团队的积极支持。忽视将导致惩罚，尽管不是直接的。

工程已将问题反馈给产品和设计部门，因为需求需要进一步澄清。产品可以决定降低优先级或者影响支持以按原始需求交付。设计部门现在期待产品部门提供澄清。产品可以即时做出决定，或者可能需要更多时间收集信息后再通知工程和设计部门下一步的行动。

猴子在哪里？在产品身上！

如果产品不能尽快解决这个问题，会导致沮丧加重，领导层因为缺乏动力会跟进。

让我们分析另一个联合问题的场景。

与上述情况相同，但工程部门回来说，他们可以按照设计提出的解决方案进行，同时满足所有产品要求，但需要重新架构系统。此外，这将为未来的需求优化系统，但会将推迟推出日期几个季度。

现在猴子在哪里？是产品部门、工程部门还是两者兼而有之？而且，是否已知这是一个共同的问题？还是因为产品部门不允许时间来重构代码和优化数据库，难以避免责备之战。

产品部门陷入困境，因为业务和客户都不关心系统的重新架构，但是对于维持技术系统以实现规模和增长至关重要。工程部门希望能够获得前进的绿灯，因为其他所有事情都需要暂停。为什么工程部门需要批准？因为业务向工程部门提供资金支持。

产品部门需要将此传达到链条上。产品部门不能单独做出决定。但是，在技术方面不涉及的领导团队在做出决定之前，需要产品部门提出带有利弊的建议。

猴子又回到了产品部门身上。

这可能不是产品部门的共享观点，但却是复杂组织设计的结果。这确实是一个共同的问题，但仍需要产品部门的推动来实现战略目标。

尽管复杂性重重，产品部门仍继续承担猴子的责任。这是你的马戏团，但不是你的职责去训练猴子。

在组织复杂性高的情况下，猴子管理是有效的。它有两个关键好处。第一个是时间管理，第二个是创建能够自主处理问题的高能动员工。

如果你明白猴子背在谁身上，你就能理解时间管理和委派的艺术。
